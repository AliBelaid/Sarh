Chapter 1

Introduction

1.1 Problem Statement

Libya's real estate sector currently relies on paper deeds issued, transferred, and verified manually across municipal offices. The lack of a central registry has produced three persistent problems: (i) title insecurity, where multiple owners may legitimately hold conflicting paper deeds for the same parcel; (ii) verification opacity, which forces buyers, banks, and courts to rely on uncertain provenance chains; and (iii) identity fragility, where there is no national digital identity that can be cryptographically bound to a property holding.

These problems are amplified by the country's historical disruptions: archive losses, multiple parallel administrative systems, and the absence of a canonical reference dataset for parcel geometry.

1.2 Research Questions

  1. How can a digital registry coexist with — rather than replace — legitimate paper deeds during a multi-year transition?
  2. What architecture supports a future national digital identity (when issued by the Libyan state) without forcing a data migration?
  3. How can deeds be made tamper-evident and machine-verifiable by any third party (bank, court, citizen)?
  4. How can polygon overlap conflicts be surfaced for officer review without blocking submissions?

1.3 Objectives

  • O1. Build a unified web + mobile platform that registers properties, issues digital ID cards (NTAG 424 DNA), and produces tamper-evident PAdES-signed PDF deeds.
  • O2. Preserve every paper-era artifact via a legacy_national_no field on every citizen and a deed reference field on every property.
  • O3. Provide a public verification page accessible to anyone with a deed code or QR scan, requiring no login.
  • O4. Enforce an append-only audit log, defended at the database level by INSTEAD OF UPDATE/DELETE triggers.
  • O5. Deliver an Arabic-first RTL experience across all citizen-facing surfaces, with Latin reserved for codes and identifiers.

Chapter 2

Methodology

2.1 Development Process

Sarh follows an incremental SDLC partitioned into thirteen delivery phases (0 through 12). Each phase has an explicit Definition of Done: code passes lint and unit tests, an end-to-end happy-path runs green, API endpoints appear in OpenAPI, permissions are reviewed against the role-permission map, the Arabic UI is sanity-checked on RTL with real Arabic content, and audit log entries are verified for every write path.

2.2 Feasibility Analysis

Technical

All major components are commodity open-source: ASP.NET Core 8 for the API, SQL Server 2019/2022 for the data store, Angular 21 + Flutter for clients, Hyperledger ACA-Py for SSI, and NTAG 424 DNA chips (Mifare ecosystem) for cards. No exotic vendor lock-in.

Operational

The reference deployment is a single Docker Compose stack on a VPS (api, web, mssql, aca-py, nginx, clamav). Backups are nightly full + hourly log-shipping for SQL Server.

Economic

Per-card cost is approximately $0.85 (NTAG 424 DNA in volume). The expensive moving parts are operator training and the survey of legacy paper deeds — not technology spend.

3.4 Constraints (CLAUDE.md)

Hard constraints — non-negotiable:
  1. Arabic-first RTL for ALL user-facing UI. Latin only for codes, IDs, technical labels.
  2. Ownership of digital identity must be re-issuable: when (if) Libya launches a national digital ID, the system must migrate without data loss.
  3. Two approved properties cannot share the same centroid; polygon overlap triggers a reviewer warning, not a hard block.
  4. Use NTAG 424 DNA only; static UID is never trusted alone — every read must verify a SUN message.
  5. Every issued PDF deed must be PAdES-signed and contain a verifiable QR.
  6. The audit log is append-only — no UPDATE or DELETE allowed at the database level.
  7. No free-text role checking in code — always go through the JSON permission map on officers.permissions.

Presentation Tier

  • Audit interceptor stamps every write
  • QuestPDF + QRCoder render the deed PDF

Data Tier

  • SQL Server 2019/2022 — local for dev, hosted for prod
  • geography type for parcel polygons (SRID 4326)
  • Numbered T-SQL migrations 000–025 in infra/mssql/migrations/
  • INSTEAD OF UPDATE/DELETE triggers protect audit_log
  • JSON columns validated by CHECK ISJSON(col) = 1

Cross-cutting Services

  • Hyperledger ACA-Py (did:sov) for VC issuance
  • nginx terminates TLS, enforces CORS, rate-limits /auth/*
  • ClamAV scans every uploaded document
  • Local filesystem at STORAGE_ROOT for documents, photos, deeds

5.4 Conventions

  • Snake_case table and column names
  • Primary keys: UNIQUEIDENTIFIER DEFAULT NEWID() — except audit_log.id which is BIGINT IDENTITY for monotonic ordering
  • Time columns: DATETIMEOFFSET(3) DEFAULT SYSDATETIMEOFFSET()
  • Auto-stamped updated_at via AFTER UPDATE trigger on every entity
  • Soft delete: is_active = 0 — never physical delete
  • Geometry: SQL Server geography (SRID 4326 implicit). Spatial methods used: STIntersects, STDistance, EnvelopeCenter(), STArea()
  • ENUMs: SQL Server has none — modeled as NVARCHAR(N) CHECK (col IN (…))
  • JSONB equivalent: NVARCHAR(MAX) with CHECK ISJSON(col) = 1

5.5 Append-Only Audit (the structural defense)

CREATE TRIGGER trg_audit_log_no_update ON audit_log
INSTEAD OF UPDATE AS
BEGIN
  RAISERROR ('audit_log is append-only', 16, 1);
END;

CREATE TRIGGER trg_audit_log_no_delete ON audit_log
INSTEAD OF DELETE AS
BEGIN
  RAISERROR ('audit_log is append-only', 16, 1);
END;

The trigger is the contract: even a privileged DBA cannot rewrite history through application code paths without first dropping the trigger — an action which is itself logged at the SQL Server engine level.

6.1 Service Layer Highlights

  • JwtTokenService — signs and validates HS256 tokens from SARH_JWT_SECRET.
  • StorageService — abstracts the local filesystem under STORAGE_ROOT; all upload paths go through this single seam.
  • ReviewService — orchestrates officer approval: persists state, invokes DeedPdfBuilder, computes SHA-256, writes to audit_log.
  • DeedPdfBuilder — QuestPDF + QRCoder; renders A4 Arabic-RTL deed with three card sections, parcel map, and verify QR.
  • VerifyService — public, no-auth resolver from deed code → public deed view (incl. SHA-256 for client integrity check).

UI/UX Design

9.1 Design Philosophy

The visual system is anchored in a "PropTech paper-card" aesthetic: warm paper background, slate ink, deliberate orange accents for action, cyan for confirmation, red for caution. We deliberately avoid green to keep the language of "approved" anchored in a neutral cool tone (cyan), which reads as verified by the system rather than positive emotion.

9.5 Internationalization

Default direction is RTL. Logical CSS properties (margin-inline-end, padding-inline-start) are used throughout so the same stylesheet flips correctly when an LTR shell (e.g. the verify page) is rendered. Validation messages, placeholders, button labels, and toast copy are all Arabic by default; the verify page additionally exposes English to support foreign banks and courts.

Chapter 12

Security & Compliance

12.1 Trust Boundaries

  1. Public ↔ API: TLS 1.2+ at nginx; rate-limited /auth/* and /properties
  2. API ↔ DB: Mutual auth via sarh_app SQL login with least-privilege grants; no db_owner
  3. API ↔ ACA-Py: Internal Docker network only; admin port not exposed
  4. API ↔ Storage: Single seam via StorageService; path traversal blocked at boundary

12.2 Authentication & Authorization

  • JWT HS256, signing key = SARH_JWT_SECRET
  • Claims: sub, citizen_id or officer_id, role, permissions (JSON), exp
  • No PII in claims — never email, name, or national number
  • Officers carry a JSON permission map; [OfficerOnly("permission.key")] filter checks the map without DB roundtrip
  • Passwords: bcrypt cost ≥ 12 (BCrypt.Net-Next)

12.3 Tamper Evidence

Every approved deed PDF is rendered by DeedPdfBuilder (QuestPDF + QRCoder), then SHA-256-hashed by ReviewService; the hash and the on-disk path are stored on properties.deed_signed_hash and properties.deed_pdf_path. On every public download via GET /verify/{code}/deed.pdf, the server reads the file, recomputes SHA-256, compares against the stored hash, and returns HTTP 409 if they diverge — this is the system's primary tamper indicator. The hash is also surfaced on the public verify page so any third party can re-verify it against their own download.

12.4 NFC Anti-Clone

NTAG 424 DNA chips emit a Secure Unique Number (SUN) message at every read — a URL with a rolling counter and a per-card MAC. The verification server rejects a SUN whose counter is below the last-seen value (replay) or whose MAC does not validate against the per-card key. A static UID alone is never treated as proof of authenticity.

12.5 Audit Log

Every write endpoint emits a row into audit_log. Rows carry actor, action, target entity, before-state, after-state, IP address, and timestamp. The table has INSTEAD OF UPDATE and INSTEAD OF DELETE triggers that raise an error — no application code path can rewrite history. Trigger removal would be visible in SQL Server's own DDL audit.

12.6 Compliance Checklist

  • All citizen-facing surfaces serve Arabic by default
  • Storage buckets / directories private by default; ACL on every read
  • JWT contains no PII
  • No secrets in repository — .env.local is gitignored
  • CORS restricted to known frontend origins
  • Rate limit on /auth/* and /properties/submit
  • ClamAV scan on every uploaded document
