## 2. لمحة معمارية (Architecture Overview)

النظام مكوّن من **أربع طبقات** كما في `architecture-diagram.svg`:

**طبقة العملاء** — تطبيق المواطن (Flutter)، بوابة المواطن (Angular ويب)، لوحة موظف السجل العقاري (Angular)، محطة إصدار الهوية الرقمية (Angular مع كاميرا وقارئة NFC)، ولوحة المسؤول العام.

**طبقة الخدمات** — REST API مبنية على ASP.NET Core 8 (.NET 8) بـ C# 12 و EF Core 8، خدمة SSI / DID (Hyperledger ACA-Py) لإصدار الـ Verifiable Credentials، خدمة ترميز NFC لبطاقات NTAG 424 DNA، وحارس مصادقة JWT (HS256) مرتبط بالهوية الرقمية.

**طبقة البيانات** — Microsoft SQL Server 2019/2022 مع نوع `geography` للإحداثيات والمضلّعات (SRID 4326)، تخزين محلي على نظام الملفات لوثائق الملكية والصور تحت `STORAGE_ROOT`، مصادقة عبر `auth_users` + bcrypt، سجل تدقيق غير قابل للتعديل عبر `INSTEAD OF UPDATE/DELETE` triggers، وSignalR للاشعارات اللحظية.

**طبقة التكاملات والأجهزة** — السجل المدني الوطني (مستقبلاً)، مصلحة المساحة، قارئات NFC (ACR122U)، طابعات بطاقات (Zebra ZC350)، بوابة SMS (ليبيانا)، وMapbox للخرائط.

## 3. حلّ الهوية الرقمية الليبية (Digital ID Solution)

بما أنه لا توجد هوية رقمية وطنية حالياً، النظام يُصدر هويته الخاصة **متوافقة مع المستقبل**:

### رقم الهوية الرقمية (Digital ID Number)
صيغة موحّدة بـ ١٦ خانة:
```
LY  -  RR  -  YYYY  -  SSSSSS  -  C
│     │      │        │           │
│     │      │        │           └─ خانة تحقق Luhn
│     │      │        └────────────── رقم تسلسلي ٦ خانات
│     │      └──────────────────────── سنة الإصدار
│     └─────────────────────────────── رمز المنطقة (طرابلس=11، بنغازي=21، …)
└───────────────────────────────────── رمز الدولة الثابت
```
مثال: `LY-11-2026-000147-3`

### بنية البطاقة الفعلية
بطاقة بلاستيكية مطبوعة بألوان العلم الليبي (شريط أعلى وأسفل) مع:
- صورة المواطن (مأخوذة من محطة الإدخال)
- الاسم الرباعي (الاسم، اسم الأب، اسم الجد، اللقب)
- تاريخ الميلاد، الجنس، مكان الميلاد
- رقم الهوية الرقمية + باركود QR
- شريحة NFC مرمَّزة (NTAG 424 DNA مع توقيع SUN Message)
- تاريخ الإصدار والانتهاء (٥ سنوات)
- ختم ذهبي بالشعار

### دورة الإصدار
موظف الإدخال يفتح محطة الإصدار، يدخل بيانات المواطن، يلتقط الصورة، يطبع البطاقة، ثم يضع البطاقة على قارئة NFC ليتم ترميز:
- رقم الهوية الرقمية
- توقيع رقمي مولّد من خادم SSI
- معرف DID للمحفظة
- بصمة الصورة (hash)

### تحقق فوري (Instant Verification)
عند تسجيل الدخول من الموبايل، التطبيق يُفعّل قارئة NFC المدمجة في الهاتف، يقرأ البطاقة، يرسل التوقيع للخادم للتحقق، فيتم تأكيد الهوية. هذا يُغني عن الباسورد ويُمكن استبداله بـ PIN ٦ خانات + بصمة.

### SSI Wallet
عند إصدار البطاقة، يتم توليد محفظة SSI للمواطن مبنية على Hyperledger Aries. المواطن يُنزّل تطبيق صَرح ويستلم محفظته. كل وثيقة عقارية تُصبح Verifiable Credential يمكن للمواطن أن يُشاركها مع جهات أخرى (بنوك، محاكم، مكاتب توثيق) بصلاحيات محدّدة.

## 4. الوحدات الوظيفية (Functional Modules)

### M1 — إدارة المواطنين والهوية الرقمية
إدخال بيانات المواطن (الاسم الرباعي، الميلاد، الصور، العنوان)، إصدار رقم الهوية، طباعة البطاقة، ترميز NFC، إنشاء محفظة SSI، تجميد / إعادة إصدار البطاقة، مراجعة سجل الإصدارات.

### M2 — السجل العقاري (Real Estate Registry)
رفع طلب تسجيل عقار من المواطن (موبايل أو ويب)، مع نوع العقار (سكني / زراعي / تجاري / حكومي)، الإحداثيات (Latitude/Longitude مع GeoJSON Polygon)، رقم القطعة، المساحة، الطول، العرض، العمق، إرفاق الوثائق التثبوتية (شهادة كوركي، عقد بيع، شهادة مساحة، صور موقع).

### M3 — مراجعة وموافقة (Review &amp; Approval Workflow)
حالات الطلب: `pending → under_review → approved | rejected | needs_clarification`. الموظف يفحص تعارض الإحداثيات (PostGIS `ST_Intersects`) ومنع وجود قطعتين بنفس الإحداثيات. نظام تعليقات بين الموظف والمواطن. سجل تدقيق لكل قرار.

### M4 — إصدار الوثيقة العقارية الرقمية (Digital Deed Issuance)
بعد الموافقة، النظام يُولّد PDF موقَّع رقمياً (PAdES) يحتوي بيانات العقار، ختم رسمي، QR للتحقق، ورقم تسلسلي. أيضاً يُصدر VC في محفظة المالك. تختلف صلاحية الوثيقة حسب نوع العقار (سكني = دائم، زراعي = ٢٥ سنة، إلخ).

### M5 — التحقق العام (Public Verification)
أي شخص (محامٍ، بنك، مشتري) يستطيع مسح QR من الوثيقة على بوابة عامة `verify.sarh.ly` ويرى ملخصاً مُختصراً بدون بيانات حساسة (رقم القطعة، حالة التسجيل، تاريخ الإصدار، اسم المالك مختصراً).

### M6 — الخرائط والـ GIS
عرض كل العقارات المسجَّلة على خريطة Mapbox مع تلوين حسب النوع، البحث الجغرافي بالقرب من نقطة، اكتشاف التداخلات بصرياً، تصدير KML لمصلحة المساحة.

### M7 — الصلاحيات والتدقيق
أدوار: `super_admin`, `registry_officer`, `id_issuer`, `citizen`, `auditor`. صلاحيات مخصّصة لكل دور عبر JSON permission map. سجل تدقيق غير قابل للتعديل لكل عملية حساسة.

### M8 — الإشعارات
SMS للحالات الحرجة (الموافقة، الرفض، إصدار البطاقة)، إشعارات Push من Supabase Realtime داخل التطبيق، بريد إلكتروني اختياري، إشعارات داخل النظام للموظفين.

## 6. تدفق المستخدم (User Flow)

### للمواطن (أول مرة)
يذهب لمكتب السجل العقاري، موظف الإدخال يفتح محطة الإصدار، يُدخل بياناته، يلتقط صورته، يطبع البطاقة ويُرمّزها بـ NFC. المواطن يُنزّل تطبيق صَرح، يضع البطاقة على ظهر الهاتف، يُسجّل PIN، تُفعَّل المحفظة. يستطيع الآن من البيت رفع طلب تسجيل عقار مع كل وثائقه.

### للموظف
يفتح بوابة الموظف بحساب موحَّد، يرى قائمة الطلبات الجديدة، يفتح طلباً، يفحص الوثائق والإحداثيات، يرى التعارضات إن وجدت، يوافق أو يرفض مع سبب. عند الموافقة يُصدر النظام تلقائياً الوثيقة الرقمية والـ VC.

### للجهة الخارجية
بنك يحتاج تأكيد ملكية العقار قبل قرض. المواطن يُشارك VC من تطبيقه عبر QR. البنك يمسح QR، يرى الوثيقة موقَّعة من صَرح، ويتأكد فوراً.

## 7. الأمان

تشفير الوثائق المرفوعة بـ AES-256، اتصالات TLS 1.3 إجبارية على بوابة nginx، فحص فيروسات على كل وثيقة عبر ClamAV، توقيع رقمي PAdES للوثائق مع SHA-256 محفوظ في `properties.deed_signed_hash` ومقارنته عند كل تنزيل عبر `verify`، NFC NTAG 424 DNA يمنع نسخ البطاقة (كل قراءة تُولّد رمز ديناميكي SUN مع عدّاد متجدّد)، نسخ احتياطي يومي مشفَّر لـ SQL Server، سجل تدقيق غير قابل للحذف عبر `INSTEAD OF UPDATE/DELETE` triggers على جدول `audit_log`، خريطة صلاحيات JSON على `officers.permissions` بدلاً من فحص أدوار نصّي.
